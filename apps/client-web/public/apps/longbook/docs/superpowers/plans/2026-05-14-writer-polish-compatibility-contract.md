# 执笔台润色优化兼容性契约

生成时间：2026-05-14  
目的：进一步确认“整章润色/自动润色”不会误改原有输出逻辑和运行逻辑。

## 1. 结论先说清

严格结论：

1. 自动润色关闭时，原有 AI续写、融合写作、批量生成、保存、导出、RAG、章节树、项目加载逻辑必须保持原样。
2. 自动润色开启时，输出逻辑会进入新分支，因为用户要求“生成出来前自动润色一遍，然后再显示一个一个字输出正文”。这属于显式开启后的新功能行为，不是原逻辑被悄悄改掉。
3. 手动点击“润色”时，润色按钮本来就会改变正文；本次只把它从“末尾片段润色”升级为“整章润色”。这是用户指定要改变的润色功能，不影响未点击润色时的写作、生成、导出。
4. 导出工程逻辑不改。导出仍然导出当前保存的正文。若用户手动或自动保存了润色版，导出自然拿到润色后的正文；导出函数本身不增加新判断、不换格式、不改排序。

一句话原则：

```text
默认状态不变；显式开关才分支；分支失败必回退；旧路径不插手。
```

## 2. 不允许改变的原有逻辑

以下逻辑在实施时列为“冻结区”，除非用户另行授权：

### 2.1 应用启动与模块加载

不改变：

1. `index.html` 的主布局。
2. 现有模块的初始化顺序语义。
3. 页面直接用 `file:///F:/长篇修改专用项目文件夹/长篇/index.html` 打开的能力。
4. 现有 IndexedDB 初始化。
5. 现有项目选择、active project、项目隔离逻辑。

允许的最小新增：

1. 增加一个无副作用的新脚本 `writer_polish.js`。
2. 该脚本加载时只挂函数到 `Modules.writer`，不主动读取 DB、不主动改 DOM、不主动改设置、不主动生成内容。

### 2.2 手动编辑与保存

不改变：

1. 用户在编辑器直接输入正文的行为。
2. `onInput()` 的字数、进度、脏状态更新逻辑。
3. `save()` 的主保存流程。
4. 章节标题、大纲、目标字数、状态保存逻辑。
5. 保存后的章节树刷新逻辑。

允许的新增：

1. 接受润色后调用已有 `save({ silent: true })` 保存当前章。
2. 章节对象追加可选元数据，例如 `lastPolishedAt`，不改变原字段含义。

### 2.3 AI续写旧路径

自动润色关闭时，不改变：

1. 原 prompt 构建顺序。
2. 原 `writer_ai` 提示词获取方式。
3. 原 RAG 注入。
4. 原 NEXUS 前缀。
5. 原融合技法注入。
6. 原前章摘要注入。
7. 原 `AI.generate(prompt, config, chunkCallback)` 流式回调方式。
8. 原 `editor.value += c` 边生成边显示。
9. 原 `_sanitizeGeneratedProse(addedRaw)` 后清洗。
10. 原 `save({ silent: true, forcePostProcess: opts.flowMode !== 'manual' })`。
11. 原 MemorySystem 工作记忆写入。

实施约束：

```text
const autoPolish = await this._isAutoPolishEnabled();
if (!autoPolish) {
  // 保留旧 AI续写代码块
  return await legacyAiWritePath();
}
// 只有这里进入自动润色新路径
```

旧路径不能为了新功能改 prompt、改温度、改 max_tokens、改保存参数、改状态机。

### 2.4 融合写作旧路径

自动润色关闭时，不改变：

1. 原融合技法上下文读取。
2. 原融合 prompt。
3. 原 `AI.generate(..., chunkCallback)` 流式写入。
4. 原生成结束后的清洗、保存、toast。

实施方式和 AI续写相同：

```text
if (!autoPolish) {
  return await legacyFusionWritePath();
}
return await autoPolishFusionWritePath();
```

### 2.5 当前章自动生成旧路径

位置：`writer_batch.js` 的 `autoWriteCurrent()`。

自动润色关闭时，不改变：

1. 覆盖确认弹窗。
2. 章节大纲长度检查。
3. 世界观、RAG、循环技法、融合技法上下文构建。
4. prompt 长度诊断。
5. 原流式接收和编辑器更新。
6. 原保存章节内容。
7. 原 RAGSystem.addDocument。
8. 原自动反推细纲和实体提取。

自动润色开启时，才把“生成后的 content”送入润色管线。

### 2.6 范围批量生成旧路径

位置：`writer_batch.js` 的 `_startScopedAutoWrite()`。

自动润色关闭时，不改变：

1. 范围选择。
2. 跳过/覆盖策略。
3. 每章 prompt 生成。
4. 每章生成保存。
5. 进度条和状态更新。
6. 后处理任务。

自动润色开启且“应用到批量生成”开启时，才每章多走一次：

```text
content = await polish(content)
```

如果润色失败，保存原始生成稿，不让批量流程卡死。

### 2.7 导出逻辑

不改变：

1. 导出工程按钮原入口。
2. 导出整本 txt 的排序、拼接、文件名、下载方式。
3. 导出当前章到沉浸阅读库的逻辑。
4. 导出的数据来源仍然是章节已保存正文。
5. 不在导出时临时润色。
6. 不在导出时自动改正文。

原则：

```text
润色是写作/生成阶段的事；
导出只负责导出已保存内容。
```

### 2.8 RAG、记忆、后处理

不改变：

1. RAG 检索入口。
2. RAG 文档添加结构。
3. MemorySystem 工作记忆入口。
4. 正文反推细纲。
5. 实体抽取。
6. 章节状态和章节树刷新。

自动润色开启后的唯一区别：

```text
后处理读取的是最终保存的润色版正文。
```

这不是后处理逻辑改变，而是正文内容来源变成了用户选择保存的最终稿。

## 3. 允许改变的逻辑边界

### 3.1 手动润色

允许改变：

1. 润色输入从末尾 4000 字变成整章正文。
2. 长章自动分块。
3. 接受润色后自动保存。
4. 记录润色状态。

不允许改变：

1. 未点击润色时的正文。
2. 未接受润色时的保存内容。
3. 点击还原后的原文。

### 3.2 自动润色开启后的 AI续写

允许改变：

1. 不再把原始 chunk 直接显示给用户。
2. 原始稿先进入后台缓冲。
3. 后台润色完成后才显示润色稿。
4. 润色稿用打字机方式显示。

不允许改变：

1. 自动润色关闭时的旧流式体验。
2. 原 prompt 构建逻辑。
3. 原保存位置。
4. 原章节归属。
5. 原项目隔离。

### 3.3 自动润色开启后的批量生成

允许改变：

1. 每章生成后自动润色。
2. 保存润色版。
3. 状态栏显示“润色中”。

不允许改变：

1. 章节生成顺序。
2. 范围选择。
3. 失败跳过策略。
4. 批量任务整体推进。

## 4. 实施结构：旧路径保留，新路径旁路

### 4.1 总体结构

所有入口采用同一个分叉规则：

```text
读取自动润色开关
如果关闭：走原代码路径
如果开启：走自动润色路径
```

这比把润色逻辑揉进原流程安全。

### 4.2 函数组织建议

新增服务层：

```text
writer_polish.js
```

只提供这些能力：

```text
_isAutoPolishEnabled()
_getPolishSettings()
_polishText()
_applyPolishRegex()
_splitChapterForPolish()
_buildPolishPrompt()
_validatePolishResult()
_renderTypewriter()
```

旧函数里只增加很薄的判断：

```text
if (await this._isAutoPolishEnabled()) {
  return this._aiWriteWithAutoPolish(...);
}
// 原逻辑不动
```

### 4.3 不做全局猴子补丁

禁止：

1. 不改写 `AI.generate`。
2. 不拦截 `DB.put`。
3. 不全局重写 `_sanitizeGeneratedProse`。
4. 不全局替换 `Modules.short.getPrompt`。
5. 不在导出层做自动润色。

原因：这些全局点会影响所有模块，风险太高。

## 5. 分阶段落地计划

### 阶段 A：只新增无副作用模块

动作：

1. 新增 `writer_polish.js`。
2. 在 `index.html` 加脚本引用。
3. 新模块加载时只挂函数，不执行逻辑。

验证：

1. 打开应用。
2. 不出现报错。
3. 原 AI续写仍旧流式输出。
4. 原导出仍可点击。

如果这个阶段都影响原功能，立即回退。

### 阶段 B：只加开关，不接生成

动作：

1. UI 增加“自动润色”开关。
2. 默认关闭。
3. 状态保存到 settings。
4. 不接入 AI续写、融合写作、批量生成。

验证：

1. 默认状态下没有任何行为变化。
2. 开关刷新后状态保留。
3. 打开开关但未接入时，不改变生成。

### 阶段 C：手动润色改整章

动作：

1. `polish()` 输入改整章。
2. 长文进入分块润色。
3. 接受后保存。
4. 还原保持原文。

验证：

1. 不点击润色，正文不变。
2. 点击润色但还原，保存内容不变。
3. 点击润色并接受，保存润色版。

这是第一个有意改变输出的阶段，但只发生在用户手动点击润色后。

### 阶段 D：AI续写接自动润色

动作：

1. 自动润色关闭时，直接执行旧路径。
2. 自动润色开启时，后台生成 raw。
3. raw 润色后打字机输出 polished。
4. 润色失败回退 raw。

验证：

1. 开关关闭，旧流式输出不变。
2. 开关开启，不显示 raw，只显示 polished。
3. 润色失败不丢稿。

### 阶段 E：融合写作接自动润色

动作和验证同 AI续写。

### 阶段 F：批量生成接自动润色

动作：

1. 增加“应用到批量生成”设置，默认可跟随自动润色，也可单独关。
2. 每章生成后润色。
3. 失败保存 raw 并记录状态。

验证：

1. 自动润色关闭，批量旧路径不变。
2. 自动润色开启，保存润色版。
3. 某章失败不影响下一章。

## 6. 回归测试矩阵

| 链路 | 自动润色关闭 | 自动润色开启 | 必须验证 |
|---|---|---|---|
| 应用打开 | 和现在一致 | 和现在一致 | 控制台无报错 |
| 手动输入 | 和现在一致 | 和现在一致 | 输入、字数、保存正常 |
| 保存章节 | 和现在一致 | 和现在一致 | 刷新后内容还在 |
| AI续写 | 原 chunk 流式输出 | raw 后台生成，polished 打字机输出 | 关闭时不调用润色 |
| 融合写作 | 原 chunk 流式输出 | raw 后台生成，polished 打字机输出 | 关闭时不调用润色 |
| 当前章自动生成 | 原逻辑 | 生成后润色保存 | 失败回退 raw |
| 范围批量生成 | 原逻辑 | 每章生成后润色 | 不弹确认框 |
| 手动润色 | 用户点击后润色整章 | 同左 | 可接受/还原 |
| 导出工程 | 原导出逻辑 | 原导出逻辑 | 不临时润色 |
| 导出整本 txt | 原拼接逻辑 | 原拼接逻辑 | 只导出已保存正文 |
| RAG 后处理 | 原逻辑 | 读取最终保存正文 | 不改 RAG 结构 |

## 7. 旧路径不变的测试方法

### 7.1 关闭开关时的函数调用断言

用浏览器控制台临时包一层 spy：

```text
记录 _polishText 是否被调用
关闭自动润色
执行 AI续写
断言 _polishText 调用次数为 0
```

同样测试：

1. `fusionWrite()`
2. `autoWriteCurrent()`
3. `_startScopedAutoWrite()`

### 7.2 关闭开关时的输出断言

临时 mock `AI.generate()` 返回固定 chunk：

```text
chunk1 = "甲"
chunk2 = "乙"
chunk3 = "丙"
```

关闭自动润色后，编辑器应按旧逻辑逐步显示：

```text
甲
甲乙
甲乙丙
```

开启自动润色后，编辑器不应显示 raw 的中间状态，而应等润色完成后显示 polished。

### 7.3 导出断言

准备三章：

```text
第1章：A
第2章：B
第3章：C
```

不管自动润色开关开还是关，只要没有改正文，导出 txt 都应仍是：

```text
第1章 A
第2章 B
第3章 C
```

自动润色不允许在导出时临时改成：

```text
润色后的 A/B/C
```

## 8. 代码审查检查清单

实施时每次改完都按这个清单审：

1. 是否改了 `AI.generate` 核心？如果改了，退回。
2. 是否改了导出函数？如果不是导出需求，退回。
3. 是否在模块加载时自动读写 DB？如果有，退回。
4. 是否让自动润色默认开启？如果有，退回。
5. 是否在自动润色关闭时仍调用 `_polishText`？如果有，退回。
6. 是否改变了旧 prompt 构建顺序？如果有，退回。
7. 是否改变了旧流式 chunk 写入？如果有，退回。
8. 是否让批量生成出现每章确认弹窗？如果有，退回。
9. 是否在润色失败时清空正文？如果有，退回。
10. 是否改了章节排序或项目过滤？如果有，退回。

## 9. 对“完全不改变”的精确定义

可以保证：

```text
自动润色关闭 + 不点击手动润色 = 原有运行逻辑和输出逻辑保持不变。
```

不能保证，也不应该保证：

```text
自动润色开启后输出仍和原来一样。
```

因为自动润色开启后的目标就是：

```text
原始生成稿 → 润色稿 → 打字机输出润色稿
```

这一定会改变最终文本内容和显示时机。这个改变必须只发生在用户明确开启自动润色之后。

## 10. 最终实施口径

后续真正改代码时，采用这个口径：

1. 先备份。
2. 先加无副作用模块。
3. 再加默认关闭的开关。
4. 手动润色只改润色入口。
5. 自动润色只在开关打开后分支。
6. 每接入一个入口，都先验证开关关闭时旧逻辑不变。
7. 导出层不接润色。
8. 出错一律回退原文或原始生成稿。

这样既能实现“每章完整润色”和“生成前自动润色后再打字机输出”，也能把原有输出逻辑和运行逻辑的影响范围压到最小、可测试、可回退。
