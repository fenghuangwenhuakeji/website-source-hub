# 执笔台整章润色与自动润色规格

生成时间：2026-05-14  
范围：只规划，不改代码。目标是让“执笔台正文”的每一章都能完整进入润色流程，并支持自动润色后再打字机式输出。

## 1. 需求拆解

用户目标可以拆成四个明确能力：

1. 手动润色：在执笔台当前章点击“润色”，必须润色整章正文，而不是只润色末尾片段。
2. 逐章执行：用户可以一章一章点润色键，检查、接受或还原每章润色结果。
3. 自动润色：开启一个“自动润色”选项后，AI 生成出的正文先在后台完成润色，再把润色后的版本显示到正文编辑器。
4. 打字机输出：自动润色模式下，用户看到的是润色后的最终正文被一个字一个字输出，而不是先看到原始生成稿再被替换。

用户提到“下面正则+提示词润色”，但本轮消息里没有附具体正则和提示词正文。因此本规格先把它设计成可插拔的“润色配方”：正则清洗规则、润色提示词、分段策略、输出校验都独立配置。等用户贴出具体正则和提示词后，直接填进配方，不需要再改整体架构。

## 2. 当前代码链路地图

### 2.1 润色按钮入口

位置：`assets/js/modules_split/writer/writer_core.js`

执笔台工具栏已有“润色”按钮，点击后调用：

```text
Modules.writer.polish()
```

这个入口适合作为手动逐章润色的主入口，不需要另起一个完全新按钮。需要改的是它背后的处理范围和流程。

### 2.2 当前手动润色逻辑

位置：`assets/js/modules_split/writer/writer_ai.js`

现状：

1. `polish()` 读取当前编辑器 `#w-editor` 的正文。
2. 从 `Modules.short.getPrompt('writer_polish')` 取润色提示词。
3. 读取全局规则 `#w-rules`、润色规则 `#w-polish-rules`、提取文风 `#w-style-extracted`。
4. 优先级为：提取文风 > 润色规则 > 全局规则，并会合并强制 M06/M07。
5. 当前实际传入模型的是 `content.slice(-4000)`。
6. 结果经过 `_sanitizeEditableProse()` 清洗后，直接替换编辑器内容，并把文字变红。
7. 显示一个确认浮层，允许“替换”或“还原”。
8. `_acceptPolish()` 只清除预览状态和调用 `onInput()`，没有明确保存、没有写入“已润色”状态。

关键问题：

1. 只润色末尾约 4000 字，不满足“整章润色”。
2. 没有长章节分块策略。
3. 润色提示词如果没有在提示词库中配置，可能拿到空模板。
4. 接受润色后没有明确保存，也没有留下可追踪的润色状态。

### 2.3 当前 AI 续写逻辑

位置：`assets/js/modules_split/writer/writer_ai.js`

`aiWrite()` 当前做法：

1. 构建续写 prompt。
2. 调用 `AI.generate(prompt, ..., chunkCallback)`。
3. 模型每返回一个 chunk，就立即追加到编辑器：

```text
editor.value += c
```

4. 生成结束后只清洗新增内容，再保存。

这个链路和“生成出来前自动润色一遍，然后再显示一个一个字输出正文”冲突。自动润色需要改成：

```text
隐藏缓冲区接收原始生成稿
→ 原始生成稿进入润色配方
→ 得到最终润色稿
→ 最终润色稿再打字机式写进编辑器
→ 保存
```

### 2.4 当前融合写作逻辑

位置：`assets/js/modules_split/writer/writer_ai.js`

`fusionWrite()` 和 `aiWrite()` 类似，也是在 callback 中直接把 chunk 追加到编辑器。若“自动润色”被定义为所有执笔台生成都生效，则融合写作也需要接入同一套后台润色流程。

### 2.5 当前批量生成逻辑

位置：`assets/js/modules_split/writer/writer_batch.js`

相关入口：

1. `autoWriteCurrent()`：生成当前章正文。
2. `_startScopedAutoWrite()`：范围批量生成多章。

现状都是边接收 chunk 边更新内容，结束后执行 `_sanitizeGeneratedProse()`，再保存章节。

如果自动润色开启，批量生成的每一章也应执行：

```text
生成当前章原始稿
→ 润色当前章整章稿
→ 保存润色后正文
→ 若当前章正在打开，则以最终稿更新编辑器
```

批量模式不建议每章都弹确认浮层，否则批量会被阻塞。它应该使用自动接受策略，并在状态栏显示“第 N 章正在润色”。

## 3. 目标体验

### 3.1 手动逐章润色

入口：执笔台工具栏现有“润色”按钮。

流程：

1. 用户打开某一章。
2. 点击“润色”。
3. 系统读取当前章完整正文。
4. 如果正文较短，单次送入润色模型。
5. 如果正文较长，按自然段或场景分块，逐块润色，再拼回整章。
6. 润色结果进入编辑器预览，保持现有红色预览或升级为差异预览。
7. 用户点击“替换/接受”，系统保存当前章并写入润色状态。
8. 用户点击“还原”，恢复原文，不保存润色版。

必须满足：

1. 不丢段落。
2. 不改变人物事实、剧情顺序、世界规则、伏笔状态。
3. 不输出标题、分析、评分、修改说明、字数统计。
4. 可以重复润色，但每次润色都以当前编辑器正文为输入。

### 3.2 自动润色

入口：在“润色”旁增加一个轻量下拉或开关。

建议交互：

```text
润色按钮
下拉菜单：
- 润色当前章
- 自动润色：开/关
- 自动范围：AI续写 / 融合写作 / 批量生成
```

最小可行版本只需要：

```text
润色当前章
自动润色：开/关
```

自动润色开启后：

1. 用户点击“AI续写”或批量生成。
2. 模型生成的原始正文不直接写入编辑器。
3. 系统在后台拿到完整原始稿。
4. 原始稿经过正则预清洗。
5. 原始稿进入润色提示词。
6. 润色结果经过正则后清洗和安全校验。
7. 最终润色稿才以打字机效果写入编辑器。
8. 系统保存润色后正文。

用户看到的状态应类似：

```text
AI生成中...
正在润色生成稿...
正在输出润色版...
已保存
```

### 3.3 自动润色失败时的保护

为了不破坏原有导出工程功能，自动润色失败时不能让生成结果消失。

失败策略：

1. 原始生成成功、润色失败：保留清洗后的原始生成稿，提示“润色失败，已保留原始生成稿”。
2. 润色结果为空：拒绝覆盖原稿，回退原始生成稿。
3. 润色结果明显过短：回退原始生成稿，并提示需要检查润色提示词或正则。
4. API 中断：保持原文或已生成原稿，不写入空内容。

## 4. 润色配方设计

新增一个逻辑概念：`PolishRecipe`。它不是必须立即做成单独数据库表，最小实现可以存在 `settings` 中。

建议结构：

```text
writer_polish_recipe
- version
- promptTemplate
- preRegexRules
- postRegexRules
- chunkSize
- chunkOverlap
- strictOutputOnly
- preserveParagraphIntent
```

### 4.1 正则预清洗

预清洗处理模型输入前或原始生成稿进入润色前的明显噪声。它只能清理元信息，不能擅自改剧情事实。

建议默认规则：

```text
去掉开头代码围栏：/^```(?:\w+)?\s*\n?/i
去掉结尾代码围栏：/\n?```\s*$/i
统一换行：/\r\n?/g -> "\n"
压缩三连空行：/\n{3,}/g -> "\n\n"
移除明显元标题行：/^(?:#+\s*)?(?:润色后|以下是|正文|输出|分析|总结|自检|评分|字数统计|修改说明)[:：].*$/gm
移除泄漏标签：/\[(?:M06|M07|NEXUS|Segment|emotion_score|hook_type|tension_level)[^\]]*\]/g
```

已有 `_sanitizeGeneratedProse()` 已经做了一部分元信息和禁词清理。新配方应复用它，避免两套规则分裂。

### 4.2 正则后清洗

后清洗处理模型输出后的结果，只做安全清理。

允许：

1. 去掉代码围栏。
2. 去掉“下面是润色结果”等开场白。
3. 去掉分析、总结、评分、字数统计行。
4. 清理多余空行。
5. 复用已有 `_sanitizeEditableProse()`。

不建议默认用正则做：

1. 批量改人物名。
2. 批量替换动作。
3. 批量重写比喻。
4. 自动转引号导致对话破损。
5. 删除整段“看起来像说明”的正文。

这些更适合交给提示词约束，或在用户提供明确正则后作为可关闭规则加入。

### 4.3 提示词核心要求

默认 `writer_polish` 提示词必须存在，即使用户没配置提示词库也能工作。

建议提示词骨架：

```text
你是长篇小说正文润色引擎。

任务：只润色输入正文，不续写、不删剧情、不总结、不解释。

硬性要求：
1. 只输出润色后的小说正文。
2. 不输出标题、章节名、分析、评分、字数统计、修改说明、自检。
3. 保留人物、地点、物品、剧情事实、时间线、伏笔状态、出场顺序。
4. 保留原段落意图；可以优化句式、节奏、动作、感官、对话锋利度。
5. 禁止把旁观视角改成上帝视角，禁止第一人称，除非原文明确就是第一人称且用户允许。
6. 对话使用中文双引号。
7. 执行下列写作规则，但不要把规则名写进正文。

[润色规则]
{{rules}}

[章节信息]
标题：{{title}}
大纲：{{outline}}

[上下文]
{{context}}

[需要润色的正文]
{{input}}
```

分块润色时增加：

```text
你现在只润色“当前片段”，前后文只用于保持衔接。
禁止输出前后文，禁止补写下一片段。
```

### 4.4 分块策略

整章正文不能再简单 `slice(-4000)`。建议策略：

1. 小于 6000 字：整章单次润色。
2. 6000 到 20000 字：按自然段分块，每块约 3500 到 5000 字。
3. 超过 20000 字：仍按段落分块，但每块带前后 300 字上下文，只输出当前块。
4. 分块时不切断对话段，不切断场景分隔符。
5. 拼接时保持原始段落顺序。

分块校验：

1. 输出非空。
2. 总长度不能低于原文的 65%，除非用户明确要求压缩。
3. 原文中明显的人名、专有名词、章节关键物品不能大量丢失。
4. 输出不能含有“分析、评分、修改说明、M06、NEXUS”等元词。

## 5. 数据与状态

### 5.1 设置项

建议新增或复用 `settings`：

```text
writer_auto_polish_enabled: boolean
writer_polish_recipe: object
writer_polish_typewriter_speed: number
writer_polish_apply_to_batch: boolean
```

默认：

```text
writer_auto_polish_enabled = false
writer_polish_typewriter_speed = 12 到 24 字/秒之间的稳定值
writer_polish_apply_to_batch = true
```

默认关闭自动润色可以保证原有生成体验不被突然改变。

### 5.2 章节元数据

章节记录可以追加非破坏字段：

```text
polishStatus: "raw" | "preview" | "polished" | "failed"
lastPolishedAt: ISO 时间
lastPolishMode: "manual" | "auto-write" | "auto-fusion" | "auto-batch"
polishRecipeVersion: string
```

不需要改变既有导出、阅读、章节树的基本字段。

### 5.3 原文恢复

手动润色：

1. 保留当前已有 `_polishOriginal`。
2. 接受后清空备份。
3. 还原时恢复备份。
4. 可选增强：把最近一次润色前正文短期保存到 `settings.writer_last_polish_backup`，防止页面刷新丢失预览。

自动润色：

1. 原始生成稿先保存在函数局部变量中。
2. 润色成功才覆盖编辑器或章节正文。
3. 润色失败时回退原始生成稿。

## 6. 代码接入点规划

### 6.1 新增润色服务层

建议新增：

```text
assets/js/modules_split/writer/writer_polish.js
```

职责：

```text
_getPolishSettings()
_getPolishRulesBundle()
_applyPolishRegex(text, stage)
_splitChapterForPolish(text, options)
_buildPolishPrompt(payload)
_polishText(text, options)
_validatePolishResult(original, polished)
_renderTypewriter(editor, text, options)
```

原因：

1. 手动润色、AI续写、融合写作、批量生成都需要同一套能力。
2. 避免继续把 `writer_ai.js` 变成超大文件。
3. 后续用户给新的正则和提示词，可以只调配方，不动生成链路。

### 6.2 手动润色改造

位置：`writer_ai.js` 的 `polish()`

改造方向：

1. 不再使用 `content.slice(-4000)`。
2. 调用 `_polishText(content, { mode: 'manual', preview: true })`。
3. 对长文自动分块。
4. 结果进入现有红色预览。
5. `_acceptPolish()` 接受后调用 `save({ silent: true })`，并写入章节润色状态。

### 6.3 AI 续写自动润色

位置：`writer_ai.js` 的 `aiWrite()`

自动润色关闭时：保持当前流式写入行为，最大程度保护原功能。

自动润色开启时：

```text
startLen = editor.value.length
rawAdded = await AI.generate(prompt, config)  // 不给 chunkCallback，或只积累不显示
cleanRaw = _sanitizeGeneratedProse(rawAdded)
polishedAdded = await _polishText(cleanRaw, { mode: 'auto-write', appendBase: existingContent })
editor.value = existingContent
_renderTypewriter(editor, polishedAdded, { append: true })
save()
```

重点：

1. 用户不能先看到 rawAdded。
2. 打字机输出的是 polishedAdded。
3. 自动润色只处理新增正文，不重写既有正文，除非用户正在用“整章润色”。
4. 如果当前章为空，最终保存整章润色版。

### 6.4 融合写作自动润色

位置：`writer_ai.js` 的 `fusionWrite()`

和 `aiWrite()` 同样接入后台缓冲。自动润色关闭时保持当前行为。

### 6.5 批量生成自动润色

位置：`writer_batch.js`

入口：

1. `autoWriteCurrent()`
2. `_startScopedAutoWrite()`

自动润色开启时，每章流程：

```text
raw = generate chapter
cleanRaw = sanitize raw
polished = polish cleanRaw with chapter title/outline/context
validated = validate polished
chap.content = validated
DB.put('chapters', chap)
```

批量生成不要弹确认浮层，避免流程卡住。状态栏应显示：

```text
第 12 章：生成完成，正在自动润色...
第 12 章：润色完成，已保存
```

## 8. 验收标准

### 8.1 手动润色验收

1. 当前章正文 8000 字，点击润色后，模型输入覆盖整章，不只取末尾 4000 字。
2. 润色结果保留原章节主要人物、物品、伏笔、时间顺序。
3. 润色结果不含标题、分析、总结、评分、字数统计、M06、M07、NEXUS。
4. 点击还原可恢复润色前正文。
5. 点击接受后保存章节，刷新页面后仍是润色版。
6. 已接受章节记录 `lastPolishedAt` 和 `lastPolishMode = manual`。

### 8.2 自动润色验收

1. 自动润色关闭时，AI续写保持现有边生成边显示。
2. 自动润色开启时，AI续写过程中不显示原始生成稿。
3. 润色完成后，编辑器以打字机方式输出润色版。
4. 输出结束后自动保存当前章。
5. 润色失败时不丢失原始生成稿。
6. 融合写作也遵循同样规则，除非设置中明确关闭。

### 8.3 批量生成验收

1. 自动润色开启时，批量生成每一章保存的都是润色后正文。
2. 批量流程不弹出每章确认框。
3. 某章润色失败时只影响该章，批量流程可继续，并在日志中标记。
4. 当前正在打开的章节如果被批量写入，编辑器显示最终润色版。

### 8.4 正则+提示词验收

1. 用户提供的正则可以进入 `preRegexRules` 或 `postRegexRules`。
2. 用户提供的提示词可以覆盖或扩展默认 `writer_polish`。
3. 正则执行顺序稳定可追踪。
4. 任意正则导致输出过短时触发回退保护。

## 9. 风险与控制

### 9.1 长章 token 风险

风险：整章太长，单次润色超出上下文。  
控制：按段落分块，并给每块有限前后文。

### 9.2 润色改剧情

风险：模型把润色当改写。  
控制：提示词明确“不续写、不删剧情、不改事实”，并加入结果校验。

### 9.3 自动润色等待时间变长

风险：生成后还要再润色，用户感觉慢。  
控制：状态栏分阶段显示；自动润色默认关闭；打字机阶段只输出最终稿。

### 9.4 批量生成成本增加

风险：每章多一次润色 API 调用。  
控制：增加“自动应用到批量生成”开关；失败时回退原始稿。

### 9.5 提示词缺省为空

风险：当前 `Modules.short.getPrompt('writer_polish')` 如果没有默认模板，润色 prompt 可能不完整。  
控制：新增代码级 fallback 模板，保证无配置也能润色。

## 10. 结论

本功能的核心不是简单加一个按钮，而是把“润色”抽成执笔台统一后处理管线：

```text
正文来源
→ 正则预清洗
→ 整章或分块润色提示词
→ 正则后清洗
→ 结果校验
→ 预览/打字机输出
→ 保存/回退
```

这样可以同时覆盖手动逐章润色、AI续写自动润色、融合写作自动润色、批量生成自动润色，并且不会破坏现有导出工程和生成链路的默认行为。
