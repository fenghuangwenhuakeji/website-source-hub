# 酒馆项目迭代更新指南

## 纲领性原则

根据纲领性文件的要求：
> "写一个requirement.MD 文件 一个 DESIGN.MD 文件 一个 task.MD文件 然后直接按照这样去继续接下去 继续不断完善即可 一轮一轮地整 一轮一轮地更新"

核心循环：
```
REQUIREMENT → DESIGN → TASK → REQUIREMENT → DESIGN → TASK → ...
```

---

## 迭代流程

### 第一轮迭代：任务状态更新

#### 步骤1: 更新 TASK.MD
根据当前项目实际完成情况，更新任务状态：

```markdown
### Task 1.1: 项目结构搭建
**状态**: ✅ 已完成
**优先级**: P0
**完成日期**: 2026-03-07

**完成情况**:
- [x] 创建项目目录结构
- [x] 设置基础HTML文件 (index.html)
- [x] 创建CSS目录和主样式文件
- [x] 创建JS目录和模块结构
- [x] 配置开发环境

**验证**: 目录结构完整，基础HTML可正常运行
```

对所有已完成的任务重复此操作。

#### 步骤2: 更新总体进度
```markdown
**项目名称**: UNIFIED_TAVERN_GAME
**当前阶段**: 开发完善
**总体进度**: 75%
```

#### 步骤3: 调整后续任务
根据实际完成情况，调整剩余任务的优先级和时间安排。

## 迭代频率

### 正常迭代
- **频率**: 每周1-2次
- **触发条件**: 完成一批功能或发现重要问题

### 紧急迭代
- **频率**: 立即
- **触发条件**: 发现严重Bug或安全问题

### 版本迭代
- **频率**: 每月1次
- **触发条件**: 完成一个重要阶段或发布新版本

## 最佳实践

### 1. 小步迭代
- 每次迭代不要包含太多变更
- 专注于一个主题或一批相关任务
- 频繁迭代比大规模重构更有效

### 2. 及时更新
- 完成功能后立即更新文档
- 不要等待"以后再补"
- 保持文档的实时性

### 3. 一致性检查
- 确保三个核心文档的一致性
- TASK中的完成度要与REQUIREMENT对应
- DESIGN的实现要与TASK匹配

### 4. 历史记录
- 重要变更要有记录
- 保留归档文档
- 便于追溯和回滚

### 5. 自动化检查
- 考虑使用脚本检查文档一致性
- 可以编写简单的验证工具
- 减少人工检查的疏漏

## 常见问题

### Q: 如果发现设计与实现不一致怎么办？
A:
1. 记录不一致的地方
2. 评估是设计需要修改还是实现需要调整
3. 更新相应的文档
4. 在下次迭代中修正

### Q: 需求变更了怎么办？
A:
1. 更新REQUIREMENT.MD
2. 评估对DESIGN.MD的影响
3. 更新TASK.MD中的相关任务
4. 通知团队成员

### Q: 任务延期了怎么办？
A:
1. 在TASK.MD中标记延期原因
2. 评估对后续任务的影响
3. 调整任务优先级和时间
4. 在下次迭代中优先处理
